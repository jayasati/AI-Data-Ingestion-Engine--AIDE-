# Test data

CSV fixtures for manually exercising the assignment's core requirements
against `/preview` (CSV parsing + normalization) and `/ai/extract` (adds
semantic mapping to the 15-field CRM schema). Upload any of these at
`http://localhost:3000/import`, or `curl -F "file=@test-data/<name>.csv"
http://localhost:4000/ai/extract`.

| File                                  | Tests                                                                                                                                                                                                                                       | What to look for                                                                                                                                                                                                                                                                                                          |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `01-clean-leads.csv`                  | Happy path                                                                                                                                                                                                                                  | Every row maps cleanly, no warnings, `crm_status`/`Source` values match the enums exactly.                                                                                                                                                                                                                                |
| `02-messy-headers-facebook-leads.csv` | Semantic mapping from unfamiliar header names (`full_name`, `email_address`, `campaign_name`, `created_time`...)                                                                                                                            | The AI should still map `full_name`→`name`, `campaign_name`→`data_source`/`description`, etc., without those exact header strings existing in the schema.                                                                                                                                                                 |
| `03-multi-value-contacts.csv`         | Multiple emails/phones in one cell (comma, semicolon, and `"and"`-separated)                                                                                                                                                                | The **first** email/phone should land in `email`/`mobile_without_country_code`; the rest should be appended to `crm_note`, never silently dropped.                                                                                                                                                                        |
| `04-missing-contact-info.csv`         | The hard skip rule: _records with neither email nor mobile must be skipped_                                                                                                                                                                 | "Ghost Lead" and "Another Ghost" have no contact info at all. **This rule lives in the not-yet-built Validation stage (Phase 5.5)** — today, `/ai/extract` will still return them with `email`/`mobile_without_country_code` as `null` rather than skipping; use this file to verify the skip rule once Validation lands. |
| `05-crm-status-variations.csv`        | `crm_status` enum mapping (`GOOD_LEAD_FOLLOW_UP`\|`DID_NOT_CONNECT`\|`BAD_LEAD`\|`SALE_DONE`)                                                                                                                                               | Near-matches ("Good Lead - Follow Up", "DNC", "SALE_DONE") should map to the correct enum; things with no confident match ("New", free text) should come back `null`, never a hallucinated/invented value.                                                                                                                |
| `06-data-source-variations.csv`       | `data_source` enum mapping (`leads_on_demand`\|`meridian_tower`\|`eden_park`\|`varah_swamy`\|`sarjapur_plots`)                                                                                                                              | Same idea — near-matches map to the enum, unrelated sources ("Google Ads", "Walk-in", "Referral...") come back `null` rather than being force-fit.                                                                                                                                                                        |
| `07-date-format-chaos.csv`            | `created_at` normalization across many real-world date formats (ISO, `DD/MM/YYYY`, `MM/DD/YYYY`, `DD-Mon-YYYY`, dotted, "Month D, YYYY")                                                                                                    | Every parseable date should normalize to one consistent, `new Date()`-parseable form; `"not a date"` and the blank cell should surface a warning, not a crash or a silently wrong date.                                                                                                                                   |
| `08-formatting-noise.csv`             | Normalization robustness: leading/trailing whitespace (including in header names), mixed-case emails, currency-formatted numbers (`$`, `₹`, Indian lakh-style commas), boolean-ish values (`Yes`/`Y`/`No`/`N`/`true`/`false`), curly quotes | Whitespace collapses, emails lowercase, numbers parse despite currency symbols/separators, curly quotes normalize to straight ones. Last row (no email/phone) is another skip-rule case.                                                                                                                                  |
| `09-malformed-ragged-rows.csv`        | CSV parser resilience: a blank row, a row with extra columns, a row with too few columns, a row of only empty cells                                                                                                                         | The parser should recover (pad/truncate) rather than fail the whole file, and flag recovered rows in the preview's warnings.                                                                                                                                                                                              |
| `10-realistic-mixed-batch.csv`        | Everything above combined in one realistic-looking CRM export                                                                                                                                                                               | Good file for an end-to-end demo/screenshot — messy headers, multi-value contact cell, two skip-rule candidates, mixed date formats, currency budgets, status/source variations, one ragged row.                                                                                                                          |

## Notes

- These are synthetic names/emails/phone numbers — not real people.
- `01`–`03` and `05`–`08` are deliberately "clean enough" (every row has at
  least an email or phone) so you can isolate one behavior at a time; `04`
  and `10` are the ones that exercise the skip rule.
- The AI extraction results depend on which provider is configured
  (`AI_PROVIDER` in `apps/api/.env`). With no `.env` / `AI_PROVIDER=mock`,
  results come from the deterministic Mock provider (pattern/header-based
  guessing) rather than a real LLM — good for checking the pipeline
  mechanics, not for judging real semantic-mapping quality. Point it at a
  real provider (see `apps/api/.env.example`) to test actual LLM judgment
  calls like the `crm_status`/`data_source` near-match mapping in files
  `05` and `06`.
